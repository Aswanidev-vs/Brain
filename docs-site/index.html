Brain

Captures AI conversations from ChatGPT, Claude, Gemini, and Grok into an Obsidian vault you own. Eight hundred lines of JavaScript, no dependencies, no network egress, no telemetry.

Overview

What it is, and what it isn't.

The popular AI chat clients give you a scrollback, a share button, and an account lock-in. They do not give you a file. Obsidian, on the other hand, is a folder — every note is a .md file on your disk. Brain is the bridge between the two. It runs in the browser, watches the chat you are having, and writes a copy of the conversation into your Obsidian vault the moment it lands.

By default it waits five seconds after the AI finishes its turn, then captures. The output is plain Markdown with YAML frontmatter — the same shape Obsidian expects for properties and tags. Optionally it also writes a companion graph note that links the conversation to its extracted keywords and topic tags, so the note shows up wired into your Obsidian graph.

Three rules shape every design decision: it is local only — no network destination other than 127.0.0.1; it is auditable — the entire source tree fits in a single reading session, with no build step and no dependencies; and it is reversible — the output is a folder of .md files you can delete, sync, version-control, or simply stop using without losing your data.

Architecture

Four contexts, one extension.

A Manifest V3 extension is not one program. It is a federation of small programs, each running in its own context with its own lifetime, communicating through an asynchronous messaging layer. Brain uses four of those contexts, and the design of the whole system is really about deciding which context owns which concern.

01

Manifest

The static contract. Tells Chrome which hosts the extension is allowed to touch, which scripts to inject where, which permissions to request, and which icon to put in the toolbar. Not code — the request for permission to run code.

manifest.json

02

Content script

An IIFE that runs inside the AI chat page's DOM. Recognises the platform, extracts the messages, and watches for new ones. Cannot make privileged calls — sends messages to the service worker instead.

content/platform.js

03

Service worker

A short-lived background script. The only context that can call the Obsidian REST API, write files via the downloads API, or read and write chrome.storage. Also generates the Markdown.

background/service-worker.js

04

Popup

A tiny HTML page that opens when you click the toolbar icon. The only surface the user directly interacts with — connection setup, vault selection, auto-capture settings, privacy toggles, manual capture.

popup/popup.{html,css,js}

The contract

The manifest.

The manifest is thirty-three lines. There is almost nothing decorative in it — every key is load-bearing, because Chrome refuses to install the extension if any of them are wrong. The permissions, the host list, the content-script matches, and the icons are all there because Chrome asks for them up front.

There are two distinct kinds of hosts in host_permissions, and it is worth keeping them mentally separated. The six https://… entries are the AI services whose DOM the content script will run on — if you add a new platform, you add a host here and a matching content_scripts entry and a new case in the PLATFORMS map inside platform.js. All three places must agree.

The two 127.0.0.1 entries are the network destinations the service worker is allowed to call. HTTP on port 27123 is the default; HTTPS on 27124 is the opt-in secure variant. Both are present because the popup lets the user pick either.

Content script

Extraction & auto-capture.

content/platform.js is a single IIFE — about 350 lines — that runs once per matched page. It does five things, in this order: identify which platform the page belongs to, define per-platform extractor functions for messages, title, and message count, insert a floating Capture to Brain button into the page, wire up manual capture (button click → send to service worker), and set up a MutationObserver that watches for new messages and schedules a debounced auto-save.

Platform detection

The early return on the last line is doing a lot of work. The IIFE quietly exits on any page that is not one of the six hosts, which means the floating button and the MutationObserver never get created on, say, google.com/search. The whole file is a no-op in the wrong place.

Per-platform extractors

Every AI chat site has its own DOM convention, and none of them is stable enough to write a single universal selector. So platform.js defines a small object per platform with three methods: messages(), title(), and getMessageCount(). The methods are written defensively — primary selector, fallback selector, heuristic on class names. Three patterns repeat:

  • The data-attribute shortcut. ChatGPT exposes data-message-author-role on every message. If a site gives you such a hook, use it — it is the only selector that won't break when CSS class names change.
  • The class-name scan. Gemini and Grok do not. You query for things like .user-query and .model-response-text, or scan class substrings like [class*="message"]. Brittle by definition, but the only option, and you write a fallback for the day the class name changes.
  • The alternating-position heuristic. If neither attribute gives you the role, fall back to "the first message is the user, the second is the assistant" — wrong for conversations that start with the AI, but the last-resort guess that almost always happens to be right.

The signature pattern

Auto-capture is harder than manual capture. The user could be scrolling, editing, regenerating, or pasting — and the only signal you have is "the DOM changed." Saving on every mutation would write the same conversation ten times in a second. Brain waits until the new state is stable, using a string signature of the most recent assistant message.

The signature is just the role and the trimmed content of the most recent assistant message. The auto-save timer arms when a new signature first appears, and the save only fires if, when the timer expires, the same signature is still the most recent one. Default delay is 5,000 ms — long enough that a normal-length AI response has finished streaming on a typical connection, short enough that a user clicking away from the tab will not lose their work.

The MutationObserver

The observer fires on three kinds of DOM change: childList (nodes added or removed), subtree (anywhere in the document), and characterData (text changes). The last one is what catches streaming AI responses — the response is rendered character-by-character, and Brain wants to notice that. The createButton() re-check on every mutation is defensive: the chat sites aggressively re-render, and the button can be detached.

Service worker

The saving pipeline.

The service worker is the only context that can talk to the Obsidian REST API. The content script hands it a conversation object over chrome.runtime.sendMessage, and the worker takes it from there. There are two things to do: turn the conversation into Markdown, and put that Markdown somewhere durable.

The message handler

The return true on the last line is the part most people miss. sendResponse is only delivered asynchronously if the listener returns true; without it, the channel closes before saveConversation resolves and the popup receives undefined.

Two save paths

Brain supports two ways of getting the file into Obsidian. The user picks in the popup.

A · Obsidian Local REST API

How: HTTP PUT to http://127.0.0.1:27123/vault/<path> with a Bearer token.

Pros: Files appear in Obsidian immediately. Vault does not have to be at a known filesystem path. Works on macOS, Linux, Windows without browser-level file permissions.

Cons: Requires the Local REST API community plugin to be installed, enabled, and configured with an API key.

B · Direct download

How: chrome.downloads.download with a data: URL of the Markdown.

Pros: Zero plugin required. Works the moment the extension is installed.

Cons: Files land in Chrome's download folder. The user is responsible for moving them into the vault.

PUT vs POST: the GET-then-decide trick

Keyword & topic extraction

The keyword extractor is intentionally dumb: lowercase, strip punctuation, drop stopwords and short words, count, take the top ten. The topic extractor matches regex patterns and labels them. Both are pure string operations, deterministic, inspectable, and fast — three properties an LLM cannot promise.

Popup

The popup.

The popup opens in a fixed 380-pixel-wide window above the toolbar icon, structured as three tabs: Setup, Settings, Privacy. A fourth hidden panel — Capture — only appears after a successful connection, because there is no point in showing a Capture button that would fail.

The connection test

A three-second timeout is the practical compromise: the Obsidian plugin is local, so the round-trip should be well under 100 ms in the happy case, but if the plugin is disabled or Obsidian is not running, you want the UI to fail visibly and quickly rather than spin forever. Connected is a derived fact, not a stored one — the popup calls testAPI on every load and trusts the network's answer.

All of this lives in chrome.storage.sync, which syncs across signed-in Chrome browsers. The cap is small (about 8 KB total) but more than enough for this set of preferences.

Recipe

Build a variation.

Brain is short enough to be a good first project, and a good template. This is a recipe. Do these things in this order and you will end up with a working extension that does something different.

  1. 01
    Make the manifest yours. Copy the folder. Change "name" and "version". Add the hosts you want to capture from into "host_permissions". For each new host, add a matching entry in "content_scripts". Three places must agree — the host list, the content_scripts.matches list, and the platform map in the content script.
  2. 02
    Write the extractor. In content/platform.js, change the PLATFORMS map. Add a case to getExtractors() with three methods — messages(), title(), getMessageCount(). Spend the time here. Inspect the target DOM, find a data-attribute you can rely on, and fall back to class-name scan only if there is no other way.
  3. 03
    Decide the output shape. The Markdown generator in service-worker.js is a pure function. Rewrite generateMarkdown to produce whatever shape you want — Notion JSON, CSV, a Daily-Journal entry, an email digest. The saving machinery does not care what string you give it.
  4. 04
    Wire a new sink (optional). To save somewhere other than Obsidian, add a third branch in saveConversation next to the api and fs branches. Read connectionMethod from storage, dispatch on it, and write the file. Inside the browser sandbox you are limited to chrome.downloads and whatever your content script can do on a privileged origin.
  5. 05
    Test it. chrome://extensions → Developer mode → Load unpacked → point at the folder. Open the target site, click the toolbar icon, complete the connection, click the floating button. Watch the service-worker console (under the extension's "Service worker" link in chrome://extensions) for the Brain: log lines.

Trust

Security & threat model.

The extension can read and modify the DOM of the six configured AI hosts; read and write chrome.storage.sync; make HTTP requests to 127.0.0.1:27123 and 127.0.0.1:27124 only; send messages between the popup, content scripts, and service worker; and trigger downloads (in the fallback save path).

It cannot make requests to any host other than the configured ones — there is no "*://*/*" pattern in the host list, and there never will be. It cannot execute arbitrary user-supplied code at runtime — no eval, no remote script tag, no Function() constructor. It cannot phone home — search the source for fetch and you will find exactly one prefix: 127.0.0.1.

An extension with content-script access to a host is, in principle, able to do anything a script on that host can do. This is true of every browser extension you have ever installed; the question is always whether you trust the code. Brain is short enough to read, and the source is the single source of truth.

Two privacy toggles in the popup close most residual concern. Exclude source URLs drops the URL of the conversation from the frontmatter and body. Anonymize AI model names replaces the platform name (ChatGPT, Claude, Gemini, Grok) with a generic letter (Model A, Model B, …) so a casual reader of your vault cannot tell which model said what.

Notes

Troubleshooting.

"Not connected"Obsidian not running, or the Local REST API plugin is disabled. Open Obsidian, enable the plugin, confirm the HTTP server is on.
API error 401Wrong API key. Open the plugin's settings, copy the key, paste it into Brain.
Capture button missing on an AI siteHost not in the manifest's content_scripts.matches. Add the host. Reload the extension.
Auto-capture not firingMaster switch off, or delay too short for the response to settle. Popup → Settings → enable auto-capture; raise the delay to 7–10s.
Saved file is emptyExtractor selectors don't match the site's current DOM. Open DevTools, find the new selector, update the matching case in platform.js.
Files land in Downloads instead of the vaultconnectionMethod is fs (or unset). Switch to the API path in the Setup tab and re-save.
